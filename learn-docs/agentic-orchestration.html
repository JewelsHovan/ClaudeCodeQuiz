Agentic Architecture & Orchestration is 27% of the exam — the single heaviest domain. 🎯 The one idea behind every question
When a requirement must hold 100% of the time, reach for something deterministic (a hook, a programmatic gate, the loop's stop_reason). When it's best-effort, a prompt instruction is fine. Most distractors are the probabilistic option dressed up to look sufficient.

02 The agentic loop

An agent is a loop, not a script. You send a request with tools defined, then branch on the response's stop_reason — execute tools and continue, or finish.

flowchart TD S([Send request + tools]) --> R{stop_reason?} R -->|tool_use| X[Execute tools] --> A[Append tool_result
as role: user] --> S R -->|end_turn| D([Return final answer]) R -->|pause_turn| S
The loop terminates on end_turn — not on parsing the text for "done".
⚠️ Exam traps — loop termination
  • Parsing natural language ("I'm finished") to decide the loop is over.
  • Arbitrary iteration caps as the primary stop mechanism (fine as a safety net, not as the control).
  • Checking for assistant text as a completion signal — Claude can emit text alongside a tool call.
The correct control is always the structured stop_reason.
Detail worth memorizing
tool_result blocks are sent back with role: "user" and must carry the tool_use_id that matches the original tool_use call.

03 Coordinator & subagents (hub-and-spoke)

For multi-agent work, a coordinator decomposes the task, delegates to specialized subagents, and aggregates results. Subagents never talk to each other.

flowchart TD C[Coordinator
decompose · delegate · aggregate] C --> S1[Search agent] C --> S2[Analysis agent] C --> S3[Synthesis agent] S1 -.isolated context.-> C S2 -.isolated context.-> C S3 -.isolated context.-> C
All communication routes through the coordinator; each subagent runs in isolated context.

Context isolation

Subagents do not inherit the coordinator's conversation history. You pass what they need explicitly in their prompt.

Parallel delegation

Emit multiple Task tool calls in a single coordinator response to run subagents concurrently.

Goals, not procedures

Tell a subagent the goal and quality bar, not step-by-step instructions — let it reason.

Dynamic routing

Invoke only the subagents a query needs — not always the full pipeline.
⚠️ The two classic mistakes
Over-narrow decomposition (splitting "AI in creative industries" into only visual-arts subtasks, missing music/writing/film) and over-provisioned subagents (handing the synthesis agent web-search tools → cross-role misuse, violating least privilege). Scope each subagent's tools to its role.

04 Hooks vs prompts — the determinism question

This is the most-tested decision in the domain. Use the failure cost to choose.

flowchart TD Q{Must it hold 100% of the time?
financial / identity / security} -->|Yes| H[Hook or programmatic gate
deterministic] Q -->|No| P[Prompt instruction
probabilistic, cheaper]
Prompt instructions have a non-zero failure rate — unacceptable when compliance is mandatory.
PreToolUse — enforce before the call
Block refunds > $500 → route to human · require customer verification before any financial op · block destructive actions without approval.
PostToolUse — normalize after the call
Convert Unix timestamps → ISO 8601 · map numeric status codes → readable strings · standardize formats across MCP tools before the model sees them.
🆕 2026 currency
Stop / SubagentStop hooks can now return hookSpecificOutput.additionalContext (≤10,000 chars) to feed Claude context and keep the turn going without being flagged as a hook error (Claude Code v2.1.163).

05 Task decomposition: fixed vs adaptive

Prompt chaining (fixed sequential)
Use when the structure is known up front. E.g. code review: per-file local analysis → cross-file integration pass. Avoids attention dilution from dumping all files at once.
Dynamic adaptive decomposition
Use for open-ended work where subtasks emerge as you learn. E.g. "add tests to a legacy codebase": map structure → find high-impact areas → prioritized plan → adapt.
Multi-concern requests
When one user message contains several asks ("fix my order and change my address"): decompose → handle each (in parallel, shared context) → synthesize one unified reply. Don't answer the first and drop the rest — a frequent distractor is "add 'address all concerns' to the prompt."

07 Worked scenario — Multi-Agent Research System

Scenario
A coordinator delegates to search, analysis, and synthesis agents. The synthesis agent keeps needing to verify a single statistic, currently round-tripping through the coordinator (+40% latency). 85% of verifications are simple fact-checks.

Best move: give the synthesis agent a scoped verify_fact tool for the simple 85%; complex verifications still route through the coordinator. This serves the common case without handing it the full web-search toolset (which would violate least privilege and invite cross-role misuse). Giving it all search tools, batching everything to the end, or pre-caching "just in case" are the tempting-but-wrong options.

08 Self-check

What signals the agentic loop should stop?
A stop_reason of end_turn. Never parse the text, count iterations as the primary control, or look for assistant text content.
A refund over $500 must always go to a human. Hook or prompt?
A PreToolUse hook — it's a financial guarantee, so it must be deterministic. A prompt instruction has a non-zero failure rate.
Your MCP tools return Unix timestamps, ISO dates, and numeric codes. Fix?
A PostToolUse hook that normalizes outputs before the model sees them — deterministic and doesn't require editing every MCP server.
Do subagents see the coordinator's conversation history?
No — isolated context. You must pass needed findings explicitly in the subagent's prompt, ideally as structured data separating content from metadata.
How do you run three subagents in parallel?
Emit multiple Task tool calls in a single coordinator response.

09 One-screen takeaways

If you remember nothing else…
Mandatory rule → deterministic (hook/gate); best-effort → prompt.
Loop stops on end_turn, never on parsed text or iteration caps.
Coordinator owns all comms; subagents are isolated and least-privileged.
Parallel = multiple Task calls in one response.
PreToolUse = enforce before; PostToolUse = normalize after.
Multi-concern → decompose, handle each, synthesize one reply.
Resume (inform of changes) · new session (stale) · fork (compare).
