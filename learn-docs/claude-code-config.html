Claude Code Configuration & Workflows

The 20% domain — CLAUDE.md hierarchy, commands vs skills, hooks, subagents, plugins, and CI workflows.

01 Why this domain pays off

Claude Code Configuration & Workflows is 20% of the exam — and it is the most mechanical domain. The questions reward people who know exactly which artifact does what: where config lives, what is shared vs personal, and which mechanism is deterministic vs probabilistic. Memorize the table of "right tool for the job" and the points fall out.

🎯 The two questions behind every item
  1. Shared or personal? Anything committed to the repo (project CLAUDE.md, .claude/rules/, .claude/skills/, .mcp.json) is shared with the team. ~/.claude/ is personal and never version-controlled.
  2. Always-on or on-demand? CLAUDE.md is always loaded; skills and commands are invoked on demand; path-specific rules load only when a matching file is touched.
Most distractors put the right content in the wrong scope, or use a probabilistic mechanism where a deterministic one (a hook, allowed-tools) is required.

02 The CLAUDE.md hierarchy & @import

Three levels stack from broadest to narrowest. All applicable levels load together; the more specific level refines the broader ones.

User-level
~/.claude/CLAUDE.md — personal preferences across all your projects. Not shared via version control.
Project-level
root CLAUDE.md or .claude/CLAUDE.md — committed to the repo, shared with the whole team, always loaded.
Directory-level
a CLAUDE.md inside a subdirectory — scoped to that directory tree. Still committed, so still shared.

personal · all projects · NOT shared"] P["Project CLAUDE.md

All applicable levels combine; the most specific scope refines the broader ones.
@import keeps it modular
Reference external files with @import ./standards/api-conventions.md instead of growing one giant monolith. Each package's CLAUDE.md imports only the standards relevant to its domain — no copy/paste, no manual sync drift.
⚠️ The classic hierarchy trap
"A new teammate clones the repo but isn't getting the conventions." The cause is almost always that the rules live in someone's ~/.claude/CLAUDE.md (personal, not shared) instead of project-level CLAUDE.md. Conversely, personal preferences (commit style, verbosity) belong in ~/.claude/CLAUDE.md so they don't leak to the team. Use /memory to see which files are actually loaded when behavior is inconsistent across sessions.

03 Path-specific rules vs directory CLAUDE.md

Both scope instructions, but they answer different questions. The deciding factor is how the relevant files are arranged: clustered in one tree, or scattered by type across the codebase.

Directory CLAUDE.md — contained in a tree
Conventions live inside one directory subtree (e.g. all Terraform under infrastructure/). Drop a CLAUDE.md in that folder.
.claude/rules/ + glob — spread by type
Conventions apply to files of a type wherever they live (e.g. test files sitting next to their source throughout the repo). Use a rule file with a paths glob.

A path-specific rule is a markdown file under .claude/rules/ with a paths glob in its frontmatter. It loads only when you edit a matching file — cutting irrelevant context and tokens.

---
paths:
  - "**/*.test.tsx"
  - "**/*.test.ts"
---
# Test Conventions
- Use React Testing Library, not Enzyme
- Test behavior, not implementation

Choose by file arrangement: contained tree vs scattered-by-type.

Glob specificity on conflict

When two rule globs match the same file, the more specific (narrower, deeper) glob wins — just like specificity decides everywhere else in the config hierarchy. Order of definition and alphabetical order are irrelevant.

File editedMatching globsGoverns
src/payments/charge.ts src/**/* vs src/payments/**/*.ts src/payments/**/*.ts (more specific)
⚠️ Trap
Conflicting rules are not both discarded, and precedence is not by definition order or alphabetical position. The narrower glob match governs.

04 Commands vs skills — the most-confused topic

This is the section to over-prepare. Commands and skills look similar (both live under .claude/, both are markdown) but their argument mechanisms and lifecycle are not interchangeable, and the exam loves to swap one's features onto the other.

Custom slash commandSkill
Lives in.claude/commands/ (project) or ~/.claude/commands/ (personal) .claude/skills/<name>/SKILL.md (project) or ~/.claude/skills/ (personal)
Takes free-form args via$ARGUMENTS substituted into the body frontmatter argument-hint (prompts for a parameter)
Context isolationruns inline in the conversation context: fork runs it in an isolated sub-agent
Tool restriction—allowed-tools limits which tools it may call
Best forquick, reusable prompt templates with user input task-specific workflows, especially verbose or tool-restricted ones
🎯 Memorize the ownership of each keyword
$ARGUMENTS belongs to COMMANDS. argument-hint, context: fork, and allowed-tools are SKILL frontmatter. If an answer attaches $ARGUMENTS to a skill, or claims context: fork captures a command's arguments, it is wrong.

Skill frontmatter you must know

---
argument-hint: "target directory"   # prompts for a required parameter
context: fork                        # isolate verbose output from the main convo
allowed-tools:                       # deterministic tool restriction
  - Read
  - Grep
  - Glob
---
# Codebase Analyzer
...verbose exploratory instructions...

context: fork

Runs in an isolated sub-agent so hundreds of lines of exploratory output never pollute the main conversation — only a summary returns. Use for codebase analysis, brainstorming.

allowed-tools

Makes write/edit tools unavailable. To guarantee a skill is read-only, restrict the toolset — don't just write "do not modify files" in prose (probabilistic).

argument-hint

Prompts the developer for a required parameter when the skill is invoked without one. It does not restrict tools or isolate context.

Skill vs CLAUDE.md

Skills are on-demand for task workflows; CLAUDE.md is always-loaded universal standards. Don't put always-on conventions in a skill (applied inconsistently).
⚠️ Personal variant of a shared skill
Want a private tweak to the team's /deploy skill without affecting teammates? Create a differently-named variant under ~/.claude/skills/ (e.g. deploy-dryrun). Do not edit the committed project skill (affects everyone) and don't try to redefine a skill via CLAUDE.md (that's not the skill mechanism).

05 Hooks — deterministic enforcement & transformation

Hooks are how you get a 100%-of-the-time guarantee that the model alone cannot give. Pick the hook by when you need to act relative to the tool call or turn boundary.

flowchart TD M[Model wants to act] --> Pre{PreToolUse} Pre -->|deny| Block["Blocked before
anything happens"] Pre -->|allow| Tool[Tool executes] Tool --> Post[PostToolUse] Post --> T["Transform / normalize
the result or file"] T --> Turn{Model ends turn?} Turn --> Stop[Stop / SubagentStop] Stop -->|additionalContext| M
PreToolUse gates before; PostToolUse transforms after; Stop can re-inject context to continue.
PreToolUse — block BEFORE it happens
Inspect the target path and deny any Edit/Write to config/secrets/. Preventive and deterministic — the change never occurs. The strongest guarantee for "this must never happen."
PostToolUse — transform AFTER it happens
Run the formatter on the file that was just written. Right place for deterministic transformations on a completed action. A PostToolUse revert is reactive (the write already happened) — weaker than blocking.
HookFiresUse it for
PreToolUsebefore a tool callblock / validate (path denylists, approval gates)
PostToolUseafter a tool calltransform / normalize (format, convert timestamps)
Stop / SubagentStopwhen a turn / subagent turn would endverify-and-continue (re-inject context, e.g. failing tests)
⚠️ Probabilistic where deterministic is required
A CLAUDE.md warning, few-shot examples, or a Stop-hook nudge all reduce but never eliminate a failure rate. When a requirement is mandatory (security, secrets, compliance), the answer is a hook (PreToolUse to block, PostToolUse to transform) or allowed-tools — never prose.

06 Subagents & context isolation

A session that wrote code is biased when asked to review its own diff — it shares the implementer's assumptions and blind spots, so it rates its own work as fine and misses real bugs.

🎯 Separate the reviewer from the implementer
The fix is architectural, not a prompt tweak: run the review in a separate reviewer subagent / instance with context isolated from the implementer. Telling the same biased session to "be a harsh reviewer," raising temperature, or re-running the same context twice all inherit the same blind spots.

This is also why CI review should use an independent instance from the one that generated the code, and why an explore subagent is used to isolate verbose discovery output and return only a summary — preserving the main context window.

07 Plugins & 2026 currency

Plugins package a team's skills, commands, and hooks so a colleague can adopt them. The 2026 workflow removed the old marketplace/install friction.

🆕 2026 currency
  • Plugin scaffolding (May 2026): claude plugin init <name> scaffolds a plugin, and SKILL.md files placed in its .claude/skills directories auto-load as <name>@skills-dir — no marketplace publish and no install step required. Manual copying into ~/.claude/skills is the old approach this replaces.
  • Stop/SubagentStop additionalContext (v2.1.163): these hooks can now return hookSpecificOutput.additionalContext (≤10,000 chars) to feed Claude context and keep the turn going without the hook being treated as an error. Example: a Stop hook checks whether tests passed and, if not, re-injects the failing output so Claude keeps fixing in the same turn — a non-zero exit, by contrast, is treated as a hook error.
  • Current model: reference Opus 4.8 as the current flagship where a model is named.

08 CI/CD: non-interactive mode & test generation

In CI there is no human at a prompt, so Claude Code must run headless.

# Non-interactive: process prompt, write to stdout, exit
claude -p "Review this diff for security issues"

# Machine-parseable findings for automated PR comments
claude -p "Review this PR" --output-format json --json-schema schema.json
⚠️ The build that hangs forever
A CI job running claude "..." without -p waits for interactive input and hangs until timeout. The fix is the -p / --print flag — not a shorter timeout, not piping yes into stdin, and there is no --interactive=false flag.

Better test generation

Provide the existing test files in context (so it avoids duplicates) and document testing standards + what counts as a valuable test in CLAUDE.md (so it skips trivial getters/setters).

Incremental reviews

Carry prior findings into context and instruct Claude to report only new or still-unaddressed issues — otherwise re-runs spam duplicate comments.

Independent reviewer

Don't let the generating session review its own code; use a fresh instance with isolated context.

Shared MCP, private tokens

Put the server in project-scoped .mcp.json (committed) and reference secrets via env expansion like ${GITHUB_TOKEN} — never commit the token.

10 Worked scenario — Monorepo config (Code Generation with Claude Code)

Scenario
A repo has Terraform under infrastructure/, React components under web/, and Python services under services/. Test files live next to the source they cover, across all three trees. You want testing conventions to apply to every test file regardless of location, while each language tree keeps its own conventions.

Best move: two complementary mechanisms.

  • Directory CLAUDE.md in each language tree — those conventions are contained within a single subtree.
  • .claude/rules/ with a glob like "**/*.test.*" for the testing conventions — they apply to files by type, spread across the codebase, and load only when a test file is touched.

A single giant root CLAUDE.md loads everything for every file (wasted context, mixed concerns); a manually-invoked skill is unreliable for always-on conventions; a root directory CLAUDE.md can't selectively target test files by type.

11 Self-check

A teammate clones the repo but isn't getting the team's coding conventions. Most likely cause?
The conventions live in someone's personal ~/.claude/CLAUDE.md (not version-controlled) instead of project-level CLAUDE.md. Use /memory to confirm which files load.
You need a command that takes a free-form arg like /triage <url>. Which mechanism?
$ARGUMENTS in the command's markdown body. That is a command feature — argument-hint and context: fork are skill frontmatter, not this.
A skill must never write files. How do you guarantee it?
Set allowed-tools to a read-only set (Read, Grep, Glob) so write/edit tools are unavailable. Prose like "don't modify files" is probabilistic. context: fork isolates context but doesn't prevent writes.
Edits to config/secrets/ must be impossible. Pre- or PostToolUse?
A PreToolUse hook that inspects the path and denies the Edit/Write before it happens. PostToolUse would let the write occur first (reactive); CLAUDE.md is probabilistic.
Two rule globs match the same file. Which wins?
The more specific (narrower/deeper) glob. Not definition order, not alphabetical.
You ran plugin init mytools and dropped SKILL.md files in its .claude/skills. What does a colleague have to do?
Nothing extra — as of May 2026 those skills auto-load as mytools@skills-dir with no marketplace publish or install step.
