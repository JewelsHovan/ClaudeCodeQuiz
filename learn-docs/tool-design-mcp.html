Tool Design & MCP Integration

The 18% domain — tool descriptions, resources vs tools, MCP error layers, server scoping, and the traps that cost points.

01 Why this domain rewards precision

Tool Design & MCP Integration is 18% of the exam and it is the most mechanical domain — the questions hinge on small, knowable facts: a default annotation value, which error field the model can see, the exact env-var syntax that fails. Get these crisp and the points are reliable. The questions cluster in the Customer Support Resolution Agent, Multi-Agent Research System, and Developer Productivity / Code Generation scenarios.

🎯 The one idea behind every question
A tool definition is a contract the model reads to decide. Selection accuracy is driven first by the description; safety is driven by host-enforced controls, not by what a tool claims about itself. When you need a guarantee, reach for something deterministic (a hook, a host policy, a structured stop_reason) — never a probabilistic prompt instruction.
MCP in one sentence
MCP is a universal connector — "USB-C for AI tools." A host (Claude Code) runs one client per server; each server is isolated and cannot see other servers or the full conversation. The host enforces all security boundaries.

02 Resources vs tools — the control model

This single distinction generates several questions. Memorize the control axis:

Resources
Application-controlled, read-only reference data the model reasons from. Catalogs, return policies, schemas, doc hierarchies. No side effects.
Tools
Model-controlled actions the model decides to invoke. Issue a refund, query a DB, send a message. Side effects.
flowchart TD Q{What does the model need?} Q -->|Read-only data
to reason from| R[Resource
app-controlled] Q -->|Do something
execute · create · modify| T[Tool
model-controlled] R --> Rx[Policies · catalogs
schemas · docs] T --> Tx[issue_refund
query_db · send_email]
Decision tree: read-to-reason → resource; act-with-side-effects → tool.
Why resources matter for cost
Exposing the SKU catalog or return policy as a resource gives the agent visibility into available data and cuts exploratory tool calls. Classic pattern: expose a database schema as a resource, and the query capability as a tool.
⚠️ Trap — modeling everything as tools
Putting read-only reference data behind tools forces wasteful exploratory calls and misuses the model-controlled primitive. And an action (issue a refund) can never be a resource — it has side effects. If a scenario lists "policy + catalog + reference tables + issue refund," the answer is resources for the three read-only items, tool for the refund.

03 Tool descriptions & annotations

Descriptions are the #1 driver of selection accuracy

The description is the primary mechanism the model uses to pick a tool. Vague or overlapping descriptions cause misrouting — the most common failure mode tested here.

Ambiguous — model can't choose
analyze_content → "Analyze content and return insights"
analyze_document → "Analyze a document and return insights"
Near-identical → misroutes ~half the time.
Differentiated — clear contract
State purpose, inputs, outputs, and when to use vs the alternative. Rename to kill overlap (analyze_content → extract_web_results); split a generic tool into extract_data_points, summarize_content, verify_claim_against_source.
When the model ignores your MCP tool
If Claude Code keeps reaching for built-in Grep instead of your search_internal_kb MCP tool, the fix is not lowering temperature, removing Grep, or moving config into CLAUDE.md. It is enhancing the MCP tool's description — explain its capabilities, what it returns, and when to prefer it over file-content search.

Annotations — know the defaults cold

Annotations are optional hints about a tool's behavior. The defaults are counterintuitive: they assume the worst case.

AnnotationDefaultMeaning
readOnlyHintfalseTool does NOT modify environment
destructiveHinttrueTool MAY perform destructive updates
idempotentHintfalseRepeat calls, same args = no extra effect
openWorldHinttrueInteracts with external entities
🔑 "Dangerous until proven safe"
Omit the annotations block entirely and the host treats the tool as potentially destructive and open-world (because destructiveHint and openWorldHint default to true). Missing annotations do NOT signal a safe, read-only tool — and they do NOT make the definition invalid (annotations are optional).
⚠️ Trap — trusting annotations for security
Annotations are hints only. A reviewer who says "we set readOnlyHint: true on cancel_subscription, so the host can auto-approve it" is wrong twice over: the hint may be misleading, and the spec states clients/hosts MUST NOT make security-critical decisions based solely on annotations. The host enforces its own authorization and confirmation policy regardless of what a tool claims.

04 The MCP two-layer error model

Probably the single most-tested MCP concept. There are two places an error can surface, and the difference is whether the model can see it.

flowchart LR C[Tool call] --> K{Failure type?} K -->|Unknown tool · bad args
server crash| P[Protocol error
JSON-RPC error field
code -32602 / -32603] K -->|API failure · timeout
business rule · denied| E[result + isError: true] P --> PM[Model does NOT see it] E --> EM[Model SEES it
can reason + recover]
Layer 1 (protocol) is hidden from the model; Layer 2 (isError in the result) is visible.

Layer 1 · Protocol errors

Unknown tool name (typo), invalid arguments, server-level failures. Go in the JSON-RPC error field. Model can't see them.

Layer 2 · Execution errors

API failures, timeouts, permission denied, business-rule violations. Go in result with isError: true. Model sees them and recovers.
// Protocol error — hidden from the model
{ "jsonrpc": "2.0", "id": 3,
  "error": { "code": -32602, "message": "Unknown tool: invalid_tool_name" } }

// Execution error — the model CAN see this
{ "jsonrpc": "2.0", "id": 4,
  "result": {
    "content": [{ "type": "text",
      "text": "DB timeout after 30s. Retry in 60s or use cached_results." }],
    "isError": true } }
⚠️ Trap — isError in the wrong field
isError: true goes in the result, never in the error field. And a business violation (refund over policy cap) is an execution error the model must see — reporting it as a protocol error hides it, so the agent can't offer the manager-approval alternative.

Structured error metadata

Beyond isError, return rich context so the caller can recover intelligently — the failure type, what was attempted, any partial results, and alternatives. A bare {"status":"error","message":"failed"} is the anti-pattern: it gives the coordinator nothing, so it blindly re-issues the identical request.

{ "isError": true,
  "errorCategory": "transient",   // transient | validation | permission | business
  "isRetryable": true,
  "content": [{ "type": "text",
    "text": "Database timeout after 30s. Retry in 60s or use cached_results." }] }
CategoryExamplesRetryable?Recovery
transientTimeout, rate-limit (429), service unavailableYesBack off & retry
validationMalformed input, missing fieldNoNeed different input
permissionAccess denied (403)NoNeed different credentials
businessPolicy violation (refund > $500)NoAlternative workflow
Batch example worth memorizing
A batch hits a 429, a malformed record, and a 403. Categorize them distinctly: 429 → transient/retryable; malformed → validation/not-retryable; 403 → permission/not-retryable. Marking all three "transient" causes endless useless retries.

Access-failure vs valid-empty-result

Conflated — silent bug
search_orders returns {"results": []} whether the DB was unreachable or the customer genuinely has no orders. The agent tells everyone "you have no orders" — even during an outage.
Distinguished — correct
DB unreachable → isError: true, transient/retryable (agent retries or escalates). Query succeeded, zero matches → a normal successful empty result the agent reports truthfully.

Subagent rule: implement local recovery for transient failures; propagate only the unresolvable errors upward, with full structured context.

05 Server scoping & env-var expansion

MCP server configuration lives in .mcp.json or ~/.claude.json — never in CLAUDE.md (that holds instructions, not server config). There are three scopes:

ScopeLocationShared?Use case
project.mcp.json (repo root)Yes — git-trackedTeam-shared servers
local (default)~/.claude.json (project-scoped)NoPersonal / sensitive / experimental
user~/.claude.json (global)NoPersonal tools across all projects
🔑 Precedence: local > project > user
To temporarily override a committed github server for yourself only, add a local-scoped entry — local beats project, and nothing about the shared file or teammates changes. User scope is the lowest precedence, so it would not override a project entry. Never edit or delete the committed .mcp.json to do this — that affects everyone.

To share servers with the whole team without committing secrets, use env-var expansion in a project-scoped .mcp.json:

{
  "mcpServers": {
    "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp/" },
    "database": {
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "${DB_CONNECTION_STRING}"],
      "env": { "DB_TIMEOUT": "30000" }
    },
    "api-server": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": { "Authorization": "Bearer ${API_KEY}" }
    }
  }
}
${VAR}
Expands to the value. FAILS if the variable is unset.
${VAR:-default}
Uses default when VAR is unset — resilient.

Expansion works in command, args, env, url, and headers. This is exactly what lets a team commit .mcp.json while each developer supplies their own credentials.

⚠️ Trap — bare ${VAR} with no default
"url": "${API_BASE_URL}/mcp" does not silently become empty and does not prompt interactively — expansion fails when the var is unset, blocking startup. The one-character-class fix is ${API_BASE_URL:-https://api.example.com}. There is no ${?VAR} or ${VAR!default} syntax. And gitignoring a hardcoded-secret .mcp.json defeats the goal — commit the file, externalize secrets with ${VAR}.

Community vs custom servers

Use community servers

Standard integrations (GitHub, Jira, Slack, PostgreSQL, Stripe) where your needs match the common case and speed matters. Code-review before production.

Build custom servers

Proprietary / niche internal systems, regulated industries (healthcare, finance), custom business logic, audit requirements, in-house security control.

Example: a HIPAA patient-records system → build custom (compliance, audit, access control). Pointing a generic community SQL server at sensitive data is a compliance anti-pattern.

06 Tool count, partitioning & client-side hooks

More tools is not more capability. Large tool sets degrade selection: schemas consume context tokens, similar tools create interference, and attention dilutes. Roughly 4–5 tools per agent is the reliability sweet spot.

The numbers (high-yield)
~50 tools → accuracy drops, 3+ s per decision, ~23k tokens per selection. Pre-filter to 3 tools → ~392 ms, ~800 tokens — a 98% token reduction, 8× speedup, and doubled accuracy.
flowchart TD N{How many tools?} N -->|< 10, misrouting| D[Fix the descriptions
differentiate · rename · split] N -->|> 15| P[Partition into subagents
~4-5 tools each] P --> CO[Coordinator delegates] D --> OK[Reliable selection] CO --> OK
Heuristic: under ~10 → fix descriptions; over ~15 → partition behind a coordinator.
⚠️ Trap — wrong remedy for the count
At 17 tools, description tweaks alone won't beat attention dilution — partition. Don't merge tools into "mega-tools" with a mode flag (hides capability), and don't pile on 17 few-shot examples (bloats context). Use a scoped cross-role tool (e.g. give the synthesis agent a narrow verify_fact for the 85% simple case, not the full web-search toolset).

Tool combination vs hook

To guarantee validate_identity runs before process_refund on every conversation, a PreToolUse hook that blocks the refund until verification has succeeded is the strongest, deterministic guarantee. Forced tool_choice only steers the first turn; a prompt rule is probabilistic; combining into validate_and_refund works but couples concerns and re-verifies even when identity is already confirmed.

PostToolUse hooks for client-side normalization/trimming

When you can't modify the upstream MCP server, a PostToolUse hook transforms results deterministically before the model sees them:

PostToolUse hook (deterministic)
Normalize mixed date formats (Unix / ISO 8601 / numeric codes) into one format · trim a 42-field lookup_order result to the 5 needed fields before it bloats context.
Probabilistic / non-fixes
System-prompt instructions and few-shot examples are probabilistic and cost tokens every turn · a bigger model still parses probabilistically · raising max_tokens accommodates the waste instead of removing it.

07 Worked scenario — Customer Support Resolution Agent

Scenario
You're designing the MCP layer for a support agent. It must reason from the return policy and SKU catalog, issue refunds (capped at $500), look up orders, and verify customer identity first. Several tools return data in inconsistent formats, and you cannot change the upstream servers.
  • Return policy + SKU catalog → resources. Read-only reference data the model reasons from; also cuts exploratory tool calls.
  • issue_refund, lookup_order, validate_identity → tools. Actions with side effects.
  • $750 refund request → process_refund returns isError: true, isRetryable: false, with a customer-friendly message pointing to the manager-approval workflow. Not transient, not a hidden protocol error, not a fake empty success.
  • Identity-before-refund guarantee → a PreToolUse hook blocking process_refund until validate_identity has succeeded — deterministic, not a prompt rule.
  • Inconsistent date formats → a PostToolUse hook normalizes outputs before the model sees them — no server changes required.
  • "You have no orders" bug → distinguish DB-unreachable (isError, transient) from a valid empty result (truthful "no matching orders").

08 Self-check

A tool omits its annotations block. How does the host treat it?
As potentially destructive and open-world — destructiveHint and openWorldHint default to true ("dangerous until proven safe"). The definition is still valid; annotations are optional.
Where does isError: true go, and who sees it?
In the result (not the error field). The model sees it and can reason about recovery. Protocol errors in the error field are hidden from the model.
A $750 refund violates the $500 cap. How should the tool report it?
isError: true, isRetryable: false, customer-friendly message pointing to manager approval. It's a business error — retrying never succeeds, so never mark it transient.
"url": "${API_BASE_URL}/mcp" and the var is unset. What happens?
Expansion fails and startup is blocked. Fix with ${API_BASE_URL:-https://api.example.com} to supply a default. It does not silently empty or prompt.
Override a committed github server for yourself only?
Add a local-scoped entry — local > project > user, so it overrides the shared entry for you only. Don't edit the committed file (affects everyone); user scope wouldn't win.
A single agent has 17 tools and misroutes. Best fix?
Partition into specialized subagents of ~4–5 tools each behind a coordinator. Description rewrites are the under-10 remedy; at 17, mega-tools and few-shots don't fix the scaling problem.
MCP tools return mixed date formats and you can't change the servers. Fix?
A PostToolUse hook that deterministically normalizes each tool's output before the model sees it. Prompt/few-shot approaches are probabilistic and cost tokens every turn.

09 One-screen takeaways

If you remember nothing else…
Resource = app-controlled read data; Tool = model-controlled action. An action is never a resource.
Tool description drives selection accuracy; differentiate, rename, split overlaps.
Annotation defaults: readOnly=false, destructive=true, idempotent=false, openWorld=true — "dangerous until proven safe."
Annotations are hints only — host MUST NOT make security decisions on them alone.
Protocol error (error field) = hidden from model; isError in result = model sees & recovers.
Categorize errors: transient (retry) · validation/permission/business (don't retry). Distinguish access-failure from valid-empty.
Scope precedence: local > project > user. Secrets via ${VAR}; resilient via ${VAR:-default} (bare ${VAR} fails if unset).
Under ~10 tools → fix descriptions; over ~15 → partition into subagents. Regulated/proprietary → custom server.
Can't change the server? PostToolUse hook normalizes/trims output deterministically.
