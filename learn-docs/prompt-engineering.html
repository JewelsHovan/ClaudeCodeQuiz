Prompt Engineering & Structured Output

The 20% domain — tool_use vs few-shot, tool_choice modes, extraction schemas, and the Batch API.

01 Why this domain matters

Prompt Engineering & Structured Output is 20% of the exam. Almost every question lives in the Structured Data Extraction scenario, with a few in Code Generation and Customer Support. The questions reward one habit: knowing exactly which mechanism gives a guarantee and which only nudges. The distractors are almost always a probabilistic technique dressed up to look like a hard guarantee.

🎯 The one idea behind every question
Constraints beat instructions. A JSON schema via tool_use guarantees valid structure; a nullable field removes the pressure to fabricate; explicit categorical criteria define a decision boundary. Prose instructions ("be conservative", "don't invent values", "return JSON") are probabilistic — and they are the wrong answer whenever a structural fix exists.
The mental model
Two error classes run through this whole domain: syntax errors (malformed JSON, wrong type, out-of-enum, missing required field) and semantic errors (right shape, wrong meaning). tool_use eliminates the first class entirely. The second class is what every other technique on this page is fighting.

02 tool_use vs few-shot — guarantee vs nudge

This is the single most confused pair in the domain, and the exam tests it directly. They solve different problems and the difference is the strength of the promise.

tool_use + JSON schema → GUARANTEES structure
Generation is constrained to conform to the schema. Malformed JSON, wrong types, missing required fields, and out-of-enum values become impossible. Use it whenever you need machine-parseable output you can rely on programmatically.
Few-shot examples → only NUDGE format
Examples raise the probability the model matches a format, but nothing enforces it — the model can still deviate on a novel document. Few-shot shines for ambiguous judgment and varied structures, not for hard structural guarantees.
⚠️ Exam trap — the false equivalence
"Add a few examples of the JSON you want" is the classic wrong answer when the requirement is guaranteed schema conformance. Few-shot makes valid output more likely; only tool_use makes invalid structure impossible. If the stem says "must always return valid structured output", reach for tool_use, not examples.

Use tool_use when…

You need reliable machine-parseable JSON, a fixed schema, enums, required/optional fields enforced at generation time.

Use few-shot when…

Format is inconsistent despite detailed instructions, or judgment calls are ambiguous (escalate vs self-serve), or document structures vary (inline citations vs bibliographies).

Few-shot best practice

2–4 targeted examples. Show the reasoning for why one choice beat the alternative — that teaches the decision boundary, not just the answer.

They compose

You can do both: tool_use for the shape, few-shot inside the prompt to steer judgment within that shape.
Worth memorizing
Few-shot generalizes to novel patterns — it's not mere pattern-matching of the cases you showed. That's why 2–4 examples covering distinct citation styles fixes inconsistent extraction across unseen variants.

tool_choice controls whether and which tool the model calls. Pick by what you need to guarantee.

⚠️ Exam trap — extended thinking incompatibility
Extended thinking is incompatible with any and forced (tool) tool_choice. If you need extended reasoning AND structured output, the supported pattern is tool_choice: auto plus extended thinking, then steer hard toward the tool via prompt and schema design. Lowering max_tokens does not resolve it — it's the mode, not the budget.
auto's catch
Under auto the model can answer in free text and skip the tool — so it does not guarantee structured output. When the stem says "always return the tool call", auto is wrong.

04 Schema design — nullable, enums, criteria

Once tool_use guarantees the shape, schema design is your main lever against semantic errors and hallucination.

Nullable & required: the #1 hallucination driver

Requiring a non-nullable field when the data is sometimes absent is the single biggest cause of fabrication in extraction pipelines. A required string slot must be filled, so the model invents a plausible value. The fix is structural, not prose.

flowchart TD D{Is this field ALWAYS
present in the source?} -->|Yes| R[required
type: string] D -->|Sometimes absent| N["nullable + optional
type: string, null
remove from required[]"] N --> NoFab[Model returns null
instead of fabricating]
Only require fields that are always present. Absent-sometimes → nullable, never required.
Drives fabrication
governing_law marked required when 40% of contracts have no such clause → the model invents jurisdictions. Adding "never invent a value" to the prompt does not fix it; the slot still demands a string.
Removes the pressure
"governing_law": {"type": ["string","null"]} and drop it from required. Now the model can answer null honestly when the clause is absent.
{
  "properties": {
    "invoice_number":  {"type": "string"},               // always present  → required
    "purchase_order":  {"type": ["string","null"]},       // sometimes absent → nullable + optional
    "governing_law":   {"type": ["string","null"],
                        "description": "Governing-law jurisdiction, or null if no clause exists"}
  },
  "required": ["invoice_number"]
}

Enum + "other" escape hatch + detail field

A closed enum forces unclassifiable documents into the nearest category, corrupting analytics. Don't drop the enum (you lose validation) and don't endlessly grow it (unmaintainable). Add an other value plus a free-text detail field captured only when other is chosen.

{
  "category": {
    "type": "string",
    "enum": ["finance", "legal", "technical", "other"]
  },
  "category_detail": {
    "type": ["string","null"],
    "description": "Required when category is 'other'. Free-text explanation of the real category."
  }
}
Why not nullable here?
Making category nullable erases the fact that the document was classifiable as "other-but-real" and discards the explanatory detail. The escape hatch keeps clean analytics on known categories and captures the rationale for the rest.

Explicit categorical criteria vs vague instructions

For classification and review, vague guidance ("be conservative", "only high-confidence findings") does not improve precision. Concrete categorical criteria — what to report vs what to skip — give the model a usable decision boundary and cut false positives.

Vague → noisy
"Be EXTREMELY conservative and only flag critical issues." No boundary; behavior barely changes; style nitpicks keep leaking through.
Explicit → precise
"Report bugs and security flaws. Skip intentional local style and minor formatting. Severity = critical when …, with a concrete code example per level." A real decision boundary.
⚠️ Trust is cross-category
A high false-positive rate in one category undermines trust in all categories. The fix: temporarily disable the noisy category to restore trust, then redefine it with explicit criteria + concrete examples before re-enabling — not "dampen every category" or "ask devs to ignore it".

Self-correction fields

Bake validation into the schema. Extract stated_total and calculated_total as separate fields plus a conflict_detected boolean so the extraction surfaces inconsistent source data for review instead of silently picking one number.

05 Error handling — syntax vs semantic

The whole repair strategy hinges on which error class you're facing.

flowchart TD G[tool_use generates output] --> SX[Syntax errors:
malformed JSON, wrong type,
out-of-enum, missing required] SX --> ELIM[Eliminated by tool_use
cannot occur] G --> SE{Semantic error?
line items != total,
value in wrong field} SE -->|Recoverable| RF[Retry with error feedback:
doc + failed output + exact error] RF --> FIX[Model self-corrects] SE -->|Data simply absent| NO[Retry cannot help —
allow null, route to human]
tool_use kills syntax errors. Semantic errors need retry-with-feedback — unless the data isn't there.
Retry-with-error-feedback pattern
  1. Extract with tool_use.
  2. Validate against schema and business rules (e.g. line items must sum to total).
  3. On failure, send a follow-up with the original document + the failed extraction + the specific validation error.
  4. The model self-corrects from concrete feedback.
⚠️ When retries can't work
Retry fixes format and placement errors. It cannot conjure data that is genuinely absent from the source — if a tax ID isn't in the document, every retry returns null. Stronger wording, lower temperature, and re-sending the identical request all fail here. The right move: allow null and route to manual handling. Note also: re-sending an identical request gives the model no new signal and reproduces the mistake.

06 Batch API — latency-tolerant only

The Message Batches API trades latency for cost on bulk, non-interactive work.

50% savings

Half price on input and output tokens.

≤ 24h SLA

Processing completes within 24 hours — no latency guarantee.

10,000 max

Up to 10k requests per batch.

29-day retention

Results retrievable for 29 days.
Use Batches
Overnight report generation · bulk document extraction · weekly tech-debt audits · content-moderation queues — anything latency-tolerant and non-blocking.
Do NOT use Batches
Blocking pre-merge CI/CD gates · user-facing interactive features · real-time chat · anything needing multi-turn tool-call loops.
⚠️ Batch gotchas the exam loves
  • Results are NOT returned in order — correlate with the custom_id you assigned each request. Never rely on array index.
  • On partial failure, resubmit only the failed custom_ids, not the whole batch.
  • No multi-turn tool-call loops within a batch request (single-turn tool use is fine).
  • A blocking, latency-sensitive workload behind a 24h SLA is always the wrong fit, even with 50% savings.
{
  "requests": [
    {
      "custom_id": "doc-001",          // YOU assign this; use it to correlate results
      "params": {
        "model": "claude-opus-4-5",
        "max_tokens": 1024,
        "messages": [{"role": "user", "content": "..."}]
      }
    }
  ]
}
Before you batch 10k
Refine the prompt on a small sample first, then run the full volume. Chunk any documents that exceed context limits.

07 Prompt structure techniques

System prompt / role
Put the persona, standing rules, and high-level task in the system prompt ("You are an invoice-extraction service…"). Sets behavior for the whole turn.
XML tags
Wrap distinct blocks in <rules>, <documents>, <example> so the model never confuses reference data for rules. The fix for "the model mixes up which block is which" in a long prompt.
Prefilling
Seed the start of the assistant turn (e.g. an opening {) to force generation to continue from that point — eliminates conversational preamble and controls the response start.
Few-shot
2–4 examples to nudge format and teach judgment (see §02). Demonstrations beat ever-longer prose.
Prefilling vs tool_use
Prefilling an opening brace controls where the response starts but does not guarantee valid JSON. When you can use tool_use, that's the stronger structured-output guarantee. Among text-generation techniques, prefilling is the one that deterministically removes preamble.
XML over alternatives
For a long prompt with role + rules + reference docs + an example, distinct XML tags beat concatenating into one paragraph (causes the confusion), forcing everything into nested JSON (awkward), or dumping it all into the system prompt (doesn't delineate the internal sections).

09 Worked scenario — Structured Data Extraction

Scenario
You extract invoice data with a tool_use tool and a strict schema. After thousands of runs you see zero malformed-JSON / wrong-type / out-of-enum errors — yet some runs put a phone number in the email field, and ~40% of invoices have no purchase order, where the model fabricates one. A separate check finds line items that don't sum to the stated total.

Diagnosis & moves:

  • Phone-in-email is a semantic error — tool_use guarantees shape, not meaning. The schema can't fix it; route via validation + retry-with-feedback if detectable.
  • Fabricated PO is the nullable-field problem: purchase_order is sometimes absent, so it must be nullable + optional, never required. That removes the fabrication pressure (prompt instructions and temperature won't).
  • Totals mismatch is a semantic/business-rule failure: extract stated_total and calculated_total + a conflict_detected flag, then retry-with-error-feedback sending the doc, the failed extraction, and the exact error.

The tempting-but-wrong options: "tighten the schema" (can't catch semantics), "add a strict instruction" (probabilistic), "lower temperature" (a required slot still must be filled), and "re-send the same request" (no new signal).

10 Self-check

You must ALWAYS get valid structured output. tool_use or few-shot?
tool_use with a JSON schema — it guarantees the structure. Few-shot only raises the probability of a format and can still deviate.
40% of contracts have no governing-law clause and the model fabricates one. Fix?
Make governing_law nullable (["string","null"]) and remove it from required. Don't rely on a "never invent" instruction or lower temperature — the required slot still forces a value.
Many schemas, unknown document type, must always return structure. Which tool_choice?
any — forces a tool call while letting the model pick the right schema. Forced (tool) would lock one schema; auto could return free text.
You need extended thinking AND guaranteed structured output. The API rejects any. What now?
Use tool_choice: auto with extended thinking — any and forced are incompatible with extended thinking — and steer hard toward the tool via prompt and schema design.
Line items don't sum to the stated total, but JSON is valid. Error class? Handling?
A semantic error (tool_use can't prevent it). Use retry-with-error-feedback: re-send the document, the failed extraction, and the specific validation error so the model self-corrects.
8,000 archived transcripts, reviewed next morning, want lower cost. Which API?
The Message Batches API — 50% savings on input and output tokens with a ≤24-hour SLA. Latency-tolerant overnight work is its ideal use case. Correlate results by custom_id.
