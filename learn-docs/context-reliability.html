Context Management & Reliability

The 15% domain — context-window tactics, graceful degradation, escalation triggers, and validation.

01 Why this domain matters

Context Management & Reliability is 15% of the exam — the smallest domain by weight, but it concentrates the questions that separate a pass from a fail. These are the "everything technically worked, but the system was still wrong" scenarios: a summary that quietly dropped the order total, a synthesis agent that skipped the middle subagent, an extraction pipeline that looked 97% accurate while one document type failed silently.

🎯 The one idea behind every question
Reliability is about not destroying information you can't recover — precise facts, source attribution, partial successes, and segment-level failures. When a distractor "simplifies" by compressing, guessing, averaging, or terminating, it is almost always wrong. The right answer preserves, annotates, or escalates instead.

Questions cluster in the Customer Support Resolution Agent, Multi-Agent Research System, Code Generation with Claude Code, and Structured Data Extraction scenarios.

02 Managing the context window

Long conversations and big explorations fill the context window. The failure mode is not just running out of space — it is which information gets lost first. Three named risks show up repeatedly.

Case facts: keep transactional data out of the summary

Progressive summarization is great at condensing prose and terrible at preserving precise values. Over many turns, "$1,249.00, ordered 2026-03-14, RMA #88231" degrades into "a recent high-value order" — and the agent then quotes wrong figures.

🔑 Case facts block
Extract transactional facts (amounts, dates, order/RMA numbers, statuses) into a persistent case facts block that travels with each prompt outside the summarized history. The values are stored verbatim, so summarization can never compress them away.
Tempting but wrong
"Disable summarization, always pass full raw history" (overflows the window) · "Instruct the model to never summarize numbers" (probabilistic, degrades over turns) · "Raise max_tokens" (affects output length, not what gets dropped).
Correct
A structured facts block, outside the part of context that gets compressed. The goal is to keep the facts that matter — not all the raw text.

"Lost in the middle"

Models reliably attend to the beginning and end of a long input but may skip content in the middle. A synthesis agent fed one long concatenation of five subagents' findings will often capture the first and last and omit the middle three.

Key findings up front

Put a key-findings summary at the beginning of the aggregated input so the salient conclusions are in a high-attention position.

Explicit section headers

Give each detailed block an explicit header so middle content is addressable rather than buried.

Not a temperature knob

Temperature controls output randomness, not attention across a long input. Reversing the order just moves the neglected content to a new middle.

Trim verbose tool outputs before they bloat context

An order-lookup tool that returns 40+ fields when only 5 matter wastes ~35 fields of context on every call. Across a long conversation those payloads accumulate and crowd out everything else.

Trim at the boundary
Strip tool output down to the fields the task needs before it enters the conversation context. A bigger context window only delays the problem and adds cost; the waste per call remains.

Context degradation in long sessions

In extended sessions, models start giving inconsistent answers and referencing "typical patterns" instead of the specific classes they discovered earlier. Four tools fight this:

  • Scratchpad files — persist key findings to disk so they survive context boundaries.
  • Subagent delegation — spawn focused subagents ("trace the refund flow") while the main agent coordinates, keeping each context small.
  • Phase summaries — summarize one exploration phase's findings before starting the next, and inject that summary into the new phase's initial context.
  • /compact — when verbose discovery output has filled the context and answers degrade, compact the conversation and continue the same session rather than restarting from scratch.
⚠️ Trap — the "just restart" / "just enlarge" reflex
Restarting from scratch discards all the discovered findings. A larger window still fills and still degrades. Re-asking the same questions reclaims no space. The fix targets the bloat: compact, externalize, delegate.

03 Reliability & graceful degradation

In multi-agent systems the dangerous failures are the quiet ones. A subagent that returns an empty result marked "success" makes the coordinator treat incomplete data as complete — and synthesis is silently corrupted.

Graceful degradation: one source failing should never terminate the whole report. Continue, and annotate the gap.

Structured error context, not generic statuses

A propagated error like {"status":"error","message":"search unavailable"} tells the coordinator nothing, so it blindly retries the same timing-out query. A propagated error must carry:

Failure type

Transient, validation, permission, or business — so the coordinator can decide whether a retry even makes sense.

What was attempted

The query and parameters, so the coordinator doesn't repeat the identical failing call.

Partial results

Whatever succeeded before the failure (e.g. 3 of 5 documents) — never discard it.

Potential alternatives

Other approaches the coordinator could try instead.

Access failure vs valid empty result

These must stay distinguishable. A timeout or service-down is an access failure — unresolved, may warrant a retry. A successful query that found zero matches is a valid empty result — correct data, no error, no retry. Conflating them either wastes retries on genuine zero-result answers or silently drops real failures.

Anti-patterns
Empty-result-as-success (silent suppression) · terminating the workflow on a single failure · unbounded retries · backfilling a gap with the model's general knowledge to "look complete."
Graceful degradation
Local recovery for transient faults first · propagate only what you can't resolve, with full context · continue with partial results · annotate which findings are well-supported vs which subtopics have gaps.

04 Escalation triggers

This is the most-confused area in the domain, and it hinges on one distinction: explicit request vs implicit frustration. They look similar in a transcript and call for opposite actions.

⚠️ Trap — multiple customer matches
When a lookup returns several accounts for "John Smith," ask for an additional identifier (email, order number, account ID). Do not guess by heuristic — "most recent activity," IP geolocation, or acting on all three in parallel all risk touching the wrong person's account.
⚠️ Trap — self-reported confidence as a routing signal
Auto-escalating "when the model's confidence score drops below a threshold" is an anti-pattern: LLM self-reported confidence is poorly calibrated — models are often confidently wrong on exactly the hard cases that should escalate. Prefer an independent review/verification instance over the model grading itself. Sentiment/frustration also does not correlate with case complexity.

05 Validation & confidence

Before you reduce human review or trust an automated pipeline, the question is always: could a good average be hiding a bad segment?

Aggregate metrics mask segment-level failures

"97% overall accuracy" can hide a document type or a field that performs far worse. Acting on the aggregate alone is the trap.

🔑 Stratified validation
Before reducing human review, validate accuracy stratified by document type AND field, using a labeled set. Use stratified random sampling of high-confidence extractions to measure the real error rate and detect novel error patterns — confirming performance is acceptable across all segments, not just on average.

Field-level confidence — calibrated, not self-graded

  • Have the model emit confidence per field, then calibrate thresholds against a labeled validation set — don't trust raw self-reported numbers.
  • Route low-confidence extractions to human review.
  • Analyze accuracy by document type and by field.

Bounded retry-with-error-feedback

When an extraction fails a semantic check (e.g. line items don't sum to the invoice total), resending the document plus the specific validation error is a strong recovery move — but it must be bounded. Some documents can never validate (the source itself is inconsistent), so unbounded retries loop forever and burn cost.

Correct
Bound the retries. When the bound is exhausted, route to human review carrying the captured error context.
Wrong
Unlimited retries "until it validates" · an arbitrary 50-failure cap (still effectively unbounded per item) · disabling validation so extraction "always succeeds" (hides real errors).

06 Provenance & structured subagent output

Source attribution is lost during summarization when findings are compressed without preserving the claim-to-source link. A report that states "battery storage costs fell 76% over the last decade" with no source is unverifiable.

🔑 Claim-source mappings
Require subagents to output structured mappings — source URL/document, relevant excerpt, and publication/collection date — that are preserved through synthesis. Re-searching to re-attach a source after the fact is unreliable; a bulk bibliography doesn't map specific claims to specific sources.

Conflicting sources

Report both values with their attribution and annotate the conflict — don't silently pick one, drop both, or average them. Let the coordinator/reader reconcile.

Temporal data

Carry dates so a difference like "45% (2022)" vs "52% (2024)" isn't misread as a contradiction.

Structured output for tight budgets

When downstream agents have small context budgets, return key facts, citations, and relevance scores with metadata — not verbose prose and reasoning chains.

Render by content type

Financial data → tables, news → prose, technical findings → structured lists. Don't force everything into one uniform format.
Report structure
A reliable synthesis explicitly distinguishes well-established findings, contested findings, and findings with gaps due to unavailable sources.

08 Exam traps at a glance

⚠️ The distractors that cost points
  • Bigger window / higher max_tokens as a fix for context bloat or lost data — delays the problem, doesn't solve it.
  • Empty-result-as-success — silent suppression; the coordinator can't tell failure from data.
  • Terminate the whole workflow on one subtopic failing — discards valid work; degrade gracefully.
  • Guess the customer by heuristic on multiple matches — ask for an identifier.
  • Route on self-reported confidence — poorly calibrated; use independent review.
  • Trust the aggregate metric — stratify by document type and field first.
  • Average / silently pick conflicting source values — annotate both with dates.
  • Unbounded retries — bound them, then escalate to human with the error context.
  • Escalate immediately on frustration OR resolve-first on an explicit human request — the two halves are swapped.

09 Worked scenario — Customer Support Resolution Agent

Scenario
A support agent handles long multi-turn chats. A customer references a $1,249.00 order and an RMA number early on. Twenty turns later the running summary has condensed those into "a recent high-value order," the order lookup returns 40+ fields per call, and the customer ends a message with "honestly, just get me a person."

Walk it through:

  1. Precise facts: the amount and RMA were lost to summarization. Fix with a case facts block outside the summary — not by disabling summarization or raising max_tokens.
  2. Tool bloat: trim the order lookup to the 5 relevant fields at the tool boundary before payloads accumulate.
  3. The closing line is an explicit request for a human → escalate immediately. No investigation first, no asking why. (If the customer had merely been frustrated while the issue was resolvable, you'd attempt resolution first and escalate only on reiteration.)

If two accounts had matched the customer's name, the agent would ask for an identifier rather than guess — and it would never route the escalation decision off a self-reported confidence score.

10 Self-check

The running summary keeps dropping the order total and RMA number. Best fix?
A case facts block holding amounts, dates, and identifiers verbatim, included in each prompt outside the summarized history. Not "disable summarization" or "raise max_tokens."
Customer's first message: "I want a human, now." The request is easy to resolve. Action?
Escalate immediately — an explicit human request is a hard trigger. Resolution-first is only for implicit frustration on a resolvable issue.
A subagent fails on 2 of 5 documents and returns empty results marked "success." Why is that wrong?
Silent suppression — the coordinator can't tell 2 failed and 3 succeeded, so synthesis treats incomplete data as complete. Return structured error context plus the 3 partial results; continue and annotate the gap.
97% overall extraction accuracy — safe to turn off human review?
No. The aggregate can mask a failing document type or field. Validate stratified by document type AND field first; reduce review only where each segment is acceptable.
Two sources report 45% (2022) and 52% (2024). How should synthesis present this?
Report both with source attribution and dates, annotating the conflict. Don't average, don't silently pick the higher/newer one — the dates may explain the difference and the reader reconciles.
You explored a codebase yesterday; 3 files changed overnight. Continue how?
Resume with --resume and explicitly tell the agent which 3 files changed for targeted re-analysis — preserving valid work while avoiding stale cached results.

11 One-screen takeaways

If you remember nothing else…
Precise facts → case facts block outside the summary.
Long input → key findings up front + explicit section headers (beat "lost in the middle").
Trim verbose tool output at the boundary; /compact beats restarting.
Partial failure → continue + annotate gaps; never terminate on one failure or fake success.
Explicit human request → escalate now. Implicit frustration → resolve first.
Multiple matches → ask for an identifier, don't guess.
Self-reported confidence is poorly calibrated → use independent review.
Stratify validation by type & field before cutting human review.
Preserve claim → source → date; annotate conflicts; bound retries then escalate.
Resume (inform of changes) · new session (stale) · fork (compare).
